## Architecture

VitalLogic is an AI-assisted naturopathic consultation platform with mandatory human expert review. Patients complete a 6-step intake form, chat with an AI to fill gaps, then a licensed practitioner reviews the AI-drafted recommendation before it's delivered.

**Stack**: TanStack Start v1 (React 19, SSR, server functions) + Vite 7 + Tailwind CSS v4 + shadcn/ui + TanStack Router/Query + Supabase (Postgres + RLS + Edge Functions) + Lovable Cloud hosting.

**Three distinct user flows**:
1. **Patient** ‚Äî `/consult` ‚Üí `/consult/$consultId` (AI chat) ‚Üí `/consult/$consultId/result` (approved view only)
2. **Expert/Practitioner** ‚Äî `/_expert/expert` dashboard: claim drafts from queue, edit, approve/reject
3. **Admin** ‚Äî `/_expert/admin/roles` and `/_expert/admin/audit`

Anonymous patients complete consults without signup; their contact email is used for access verification. Authenticated patients auto-claim consults to their history.

## Key Architectural Points

**Routing** ‚Äî File-based via TanStack Router. `src/routeTree.gen.ts` is auto-generated; never edit it directly. Route guards live in layout routes: `_authenticated.tsx` (auth) and `_authenticated/_expert.tsx` (role check).

**Authorization** ‚Äî Row-Level Security on all tables. Roles stored in `user_roles` table via a security-definer `has_role()` Postgres function. Auth context + role helpers are in `src/lib/auth.tsx`.

**Vite config** ‚Äî Uses `@lovable.dev/vite-tanstack-config` which bundles multiple plugins internally. Do NOT add tanstackStart, viteReact, tailwindcss, tsConfigPaths, or cloudflare as separate plugins ‚Äî they're already included.

**Supabase types** ‚Äî `src/integrations/supabase/types.ts` and `src/integrations/supabase/client.ts` are auto-generated by Lovable. Do not edit them.

**Tailwind** ‚Äî Config is defined inline in `src/styles.css` via `@theme` (Tailwind v4 style), not in a separate `tailwind.config.*` file. Design tokens use oklch colors with a "mystical golden lotus + circuit board" theme: deep indigo backgrounds, glowing gold + violet accents. Display font: Cormorant Garamond (serif). Body: Inter (sans).

**Path alias** ‚Äî `@/*` maps to `./src/*`.

## LLM Integration

Edge functions use a multi-provider abstraction at `supabase/functions/_shared/llm.ts`. The active provider is controlled by the `LLM_PROVIDER` env var in `supabase/.env`:

| `LLM_PROVIDER` | Default model | Credential |
|---|---|---|
| `lovable` (default) | `google/gemini-3-flash-preview` | `LOVABLE_API_KEY` |
| `anthropic` | `claude-3-5-sonnet-latest` | `ANTHROPIC_API_KEY` |
| `ollama` | `ayur` (local) | `OLLAMA_BASE_URL` |

Two edge functions consume this:
- `supabase/functions/consult-chat/` ‚Äî Streaming SSE chat (public, anon-accessible)
- `supabase/functions/generate-prescription/` ‚Äî Tool-calling draft generation (public, anon-accessible)

The Anthropic provider translates its SSE events into OpenAI format so the client code is provider-agnostic.

## Safety & Compliance

Before the `generate-prescription` edge function runs, an auto safety filter parses the intake for: **pregnancy, breastfeeding, under-18, hyperthyroid, autoimmune, current medications**. Contraindicated products are excluded from the AI's candidate set ‚Äî this is a pre-filter, not a post-filter. The full audit is persisted under `draft.safety_filtered` on the prescription row.

The patient result page surfaces a "Safety filter applied" disclosure block. The expert review modal shows the same audit so practitioners see exactly what was removed.

## Data Model Notes

**Roles** ‚Äî `app_role` enum: `user` / `expert` / `admin`. Always stored in `user_roles` table, never on `profiles`. Every grant/revoke is logged in `role_audit_log` via a `SECURITY DEFINER` trigger (immutable from client). `has_role()` security-definer function used in all RLS policies to avoid recursive RLS.

**Materia medica staging gate** ‚Äî `certified_materia_medica` rows have an `import_status` of `pending_review` or `live`. Only `live` rows are queryable for prescription drafts. Unverified SKUs can never reach patient recommendations.

**Marketplace sources** ‚Äî configured via `MARKETPLACE_SOURCES` map in the `marketplace-import` edge function. Two sources: Healthy Habitat Market (`source_authority: clinical`) and Isha Life AU (`source_authority: consecrated`). Each source has its own `defaultSourceAuthority` baked in.

**Wisdom protocols** ‚Äî `wisdom_sources` and `wisdom_protocols` tables hold the Anand Yoga corpus (practices, indications, contraindications, outcomes). Prescription drafts can attach protocols alongside products. Attached protocols are snapshotted at approval time (immutable).

**Anonymous consults** ‚Äî `anonToken` is stored in `localStorage` and sent with requests; `anon_token_hash` (hashed) is what's persisted server-side. On signup, existing anon consult + prescription transfer to the new account automatically.

All consult access (start / read / saveContact / claim / unlock) flows through the single `consult-access` edge function ‚Äî it's the canonical entry point for consult access rules.

## Product Context

- TGA (Australian Therapeutic Goods Administration) product registry integration is in `src/lib/tga.ts`
- Recommendations are called "prescriptions" internally (`prescriptions` table); the UI surfaces them as "recommendations" to patients
- All prescription state changes are written to `prescription_audit` for compliance
- The app is built in vertical "slices" (schema ‚Üí server ‚Üí UI ‚Üí safety) ‚Äî each slice ships one full thread end-to-end before the next begins
- The app is deployed at https://vitallogic.lovable.app via Lovable's CI
